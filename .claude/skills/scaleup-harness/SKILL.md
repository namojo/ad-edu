# Scaleup Harness — 브랜드 스케일업·커머스 하네스 (계층적 위임)

파트너 브랜드를 D2C 커머스(유통사를 거치지 않고 자사몰 등에서 소비자에게 직접 파는 방식)로 키우는 큰 프로젝트를, 총괄(커머스 전략가)이 **제품·페이지·유통·런칭 네 덩어리로 계층 분해(계층적 위임)** 해 전문 팀원에게 위임하고, 돌아온 산출물을 **하나의 스케일업 런칭 플랜으로 통합**하는 오케스트레이터(팀장)입니다.

**핵심 개념:** 큰 프로젝트는 한 번에 시키지 않고 계층으로 쪼갠다. 총괄이 큰 목표를 하위 목표로 나누고, 각 덩어리를 담당에게 맡긴 뒤, 다시 합쳐 마진과 정합성을 검산한다.

**실행 모드:** 하이브리드. 성격이 다른 Phase마다 모드를 섞습니다. 모든 에이전트는 `model: opus`로 호출합니다.

| Phase | 모드 | 이유 |
|---|---|---|
| Phase 3 (네 덩어리 분해·실행) | 에이전트 팀 | 제품·페이지·유통·런칭이 서로 참조하며(가격↔할인, 페이지↔라인) 합의 필요 |
| Phase 4 (문서 패키지) | 서브 에이전트 | doc-producer가 독립적으로 런칭 플랜 문서를 산출 |

## 팀 구성 — 계층적 위임 (총괄 → 하위 팀)

총괄이 위에서 전체를 보며 네 덩어리를 하위 팀에 나눠 맡깁니다. 필요 없는 팀원은 부르지 않습니다.

| 팀원(에이전트) | 계층 | 맡는 덩어리 | 사용 매뉴얼(스킬) |
|---|---|---|---|
| commerce-strategist | **총괄** | 분해·위임·통합 (전체 지휘) | commerce-strategy |
| researcher | 하위 | 시장·경쟁·채널 수수료·트래픽 조사 | market-research |
| copywriter | 하위 | 상세페이지 원고(스토리→USP→FAQ→전환) | ad-copywriting |
| media-planner | 하위 | 4주 런칭 프로모션·미디어 캘린더 | media-planning |
| doc-producer | 하위(필수) | 통합 런칭 플랜 문서화 | doc-automation |

> 기본 조합: commerce-strategist(총괄) + researcher(제품·유통 조사) + copywriter(상세페이지) + media-planner(런칭 캘린더) + doc-producer(문서). 브랜드 성격이 다르면 하위 팀을 조정한다(예: 리뉴얼이면 라인 조사 축소).

## Phase 0: 컨텍스트 확인 (초기/후속/부분 재실행 판별)

1. `_workspace/` 존재 여부 확인.
2. 실행 모드 결정:
   - **없음** → 초기 실행. Phase 1로.
   - **있음 + 재실행**(예: "신규 맛으로 다시", "이 브랜드로 다시") → 새 프로젝트 실행. 기존 `_workspace/`를 `_workspace_{YYYYMMDD}/`로 보관하고 새 브리프로 재실행하되, **하네스 구조(팀·문서 템플릿)는 재사용**한다.
   - **있음 + 부분 수정**(예: "상세페이지만 다시", "유통 채널만 보완", "런칭 캘린더만 갱신") → 부분 재실행. 해당 덩어리만 다시 위임하고 통합본만 갱신한다.
3. **변경 이력**을 `course8-scaleup/사례/CHANGELOG.md`에 남긴다(재현·회귀 방지). 진화 요청 시 이 이력을 참조한다.

## Phase 흐름 (분해 → 위임 → 통합 → 문서)

### Phase 1: 준비 — 목표 분해

1. 사용자 브리프(브랜드·스케일업 목표·기간·예산)를 총괄이 **네 덩어리로 분해**한다: 제품(라인) / 페이지(상세) / 유통(채널) / 런칭(캘린더).
2. 각 덩어리에 담당 팀원을 배정하고, 패턴(계층적 위임)을 확정한다.
3. 입력 브리프를 `_workspace/00_input/`에 저장하고, 팀 표준 문서 템플릿을 지정한다(포맷 난립 방지).

### Phase 2: 팀 구성

1. Phase 3용 `scaleup-team`을 만들고 commerce-strategist(총괄) + 배정한 하위 팀원 + doc-producer를 `model: opus`로 등록.
2. 작업을 계층적으로 등록: 상위 "스케일업 런칭 플랜" 아래 제품→(페이지·유통 병렬)→런칭 하위 태스크를 `depends_on`으로 연결.

### Phase 3: 네 덩어리 분해·실행 (에이전트 팀)
**실행 모드:** 에이전트 팀
1. researcher가 시장·경쟁과 채널 수수료·트래픽을 조사 → `_workspace/1_researcher_market-channel.md`.
2. commerce-strategist가 조사를 Read해 **라인 확장 기획**을 만들고 → `_workspace/2_commerce-strategist_lineup.md`, **유통 채널 전략표**를 확정 → `_workspace/2_commerce-strategist_channel.md`.
3. copywriter가 라인 기획을 Read해 **상세페이지 원고** 집필 → `_workspace/3_copywriter_detailpage.md`.
4. media-planner가 라인·채널을 Read해 **4주 런칭 프로모션 캘린더** 설계 → `_workspace/3_media-planner_calendar.md`.
5. commerce-strategist가 네 산출물을 **통합**하고 마진(판매가−원가−수수료−물류)·정합성(가격↔할인, 페이지↔라인, 채널↔주차)을 검산 → `_workspace/4_commerce-strategist_scaleup-plan.md`.
6. 팀원 간 상충·수치 불일치(예: 할인율이 마진을 깨뜨림)는 SendMessage로 확인, **삭제하지 말고 출처 병기**로 합의.
7. 팀을 `TeamDelete`로 정리(모드 전환 전 필수).

### Phase 4: 문서 패키지 산출 (서브 에이전트)
**실행 모드:** 서브 에이전트
1. doc-producer를 Agent 도구로 호출해 Phase 3 통합 플랜을 입력으로 준다.
2. 팀 표준 템플릿으로 제안·발표용 문서를 산출한다(예: 런칭 플랜 PDF/PPT, 유통·프로모션 표 Excel).
3. 산출 후 파일 손상·빈 파일 여부를 확인하고 실패분만 재생성한다.

### Phase 5: 검수·마무리
1. commerce-strategist(또는 qa-reviewer)가 마진 음수 채널·상세페이지 과장/허위 표현을 최종 점검 → 문제 시 해당 단계 1회 재생성.
2. 최종 산출물을 지정 경로 `course8-scaleup/사례/`로 내보낸다.
3. 변경 이력을 `CHANGELOG.md`에 기록하고, **다른 브랜드에 재실행 가능한 재사용 템플릿**으로 정리한다.
4. `_workspace/` 보존.

## 데이터 전달

- **팀 모드(Phase 3):** 태스크 + 파일 + 메시지. 하위 태스크는 상위 태스크에 `depends_on`으로 연결(제품→페이지·유통→런칭→통합).
- **서브 모드(Phase 4~5):** Agent 도구로 doc-producer 호출, 입력은 `_workspace/` 파일 경로로 전달, 산출도 파일로 회수.
- **파일 규약:** `_workspace/{phase}_{agent}_{artifact}.{ext}`.
- **최종 산출물만** `course8-scaleup/사례/`로 내보낸다.
- **모드 전환 규칙:** 팀 → 서브 전환 시 반드시 `TeamDelete` 후 Agent 호출. 서브의 파일 산출물을 이후 단계에 경로로 전달.

## 에러 핸들링

| 상황 | 대응 |
|---|---|
| 큰 목표를 한 번에 처리 시도 | 제품·페이지·유통·런칭 네 덩어리로 먼저 분해한 뒤 위임 |
| 채널 수수료·시장 근거 없음 | 리서처에게 1회 위임 확보 → 그래도 없으면 "가정" 라벨로 진행하고 근거 공백 명시 |
| 산출물 상충(할인율이 마진을 깨뜨림 등) | 삭제 금지, 출처·두 값 병기 후 총괄이 조정 이유 명시 |
| 채널 마진이 음수 | 통합 중단, 어긋난 채널·할인율 표시해 재조정(팔수록 손해인 플랜 통과 금지) |
| 상세페이지 효능·성분 과장/허위 | "검수 필요" 라벨 후 사람 검수로 반드시 넘김 |
| 문서 1개 포맷 실패 | 해당 포맷만 1회 재생성 → 실패 시 누락 명시하고 나머지 제공 |
| 팀→서브 전환 누락 | TeamDelete 없이 Agent 호출 금지(세션당 1팀) |
| 진화 이력 미기록 | CHANGELOG에 변경 이력 필수 기록(재현·회귀 방지) |

## 팀 크기

**대규모(가변, 4~5명).** 계층적 위임이므로 프로젝트 복잡도에 따라 늘어난다. 총괄 1(필수) + 하위 전문가 2~3명(제품·페이지·유통·런칭에서 선택) + doc-producer 1(필수)가 상한 가이드. 필요 없는 전문가는 부르지 않는다.

## 테스트 시나리오

### 정상 흐름
1. 사용자가 "제로톡을 D2C 커머스로 스케일업" 요청, 목표(첫 구매 6,000건·재구매율 20%) 제공.
2. Phase 1에서 총괄이 제품·페이지·유통·런칭 네 덩어리로 분해, 하위 팀 배정.
3. Phase 3 팀 모드로 라인 기획→상세페이지·유통(병렬)→런칭 캘린더 산출, 총괄이 통합·마진 검산.
4. TeamDelete 후 Phase 4 서브 모드로 doc-producer가 런칭 플랜 문서 생성.
5. 마진 음수 채널·상세페이지 표현 최종 점검 통과 → `course8-scaleup/사례/`로 내보냄, CHANGELOG 기록.
6. "신규 맛 하나 더 추가해서 다시" 재요청 시 하네스·템플릿 재사용해 제품·페이지 덩어리만 재위임.

### 에러 흐름
1. Phase 3에서 유통 채널 통합 중 편의점 채널의 캔당 남는 이익이 음수로 계산됨.
2. 총괄이 통합을 멈추고 편의점 매입가·프로모션 할인율을 표시해 media-planner·리서처에게 1회 재조정 요청.
3. 재조정 후에도 음수면 "편의점 채널 마진 미확보" 명시하고 자사몰·스마트스토어·쿠팡 3채널로 축소한 플랜 제공.
4. 상세페이지에 "무조건 살 빠짐" 표현 발견 시 "검수 필요" 라벨로 사람 검수 안내, 최종 보고에 리스크 명시.

## 실습 트리거 대표 프롬프트

```text
제로톡(제로 슈거 스파클링 음료)을 D2C 커머스 브랜드로 스케일업하는 런칭 플랜을 만들어줘. 총괄(커머스 전략가)이 큰 목표를 신규 맛 2종 라인 확장 기획, 상세페이지 원고(스토리→USP→FAQ→전환), 유통 채널 전략(자사몰·스마트스토어·쿠팡·편의점을 수수료·트래픽·타겟 적합도·운영 부담으로 비교), 4주 런칭 프로모션 캘린더로 나눠 맡기고, 다시 하나의 런칭 플랜으로 통합해줘. 모든 수치는 '예시 데이터'로 라벨을 달아줘.
```

후속 예: "신규 맛 하나 더 추가해서 다시" · "상세페이지만 감성 톤으로 다시" · "유통 채널에 자사 앱 추가해서 보완" · "런칭 캘린더를 6주로 늘려줘".
