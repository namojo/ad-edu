# 문서 자동화 매뉴얼

하네스 팀원들이 만든 콘텐츠(조사 결과·카피·미디어 플랜 등)를 **클라이언트에게 낼 수 있는 문서**로 산출하는 업무다. 이 매뉴얼은 직접 파일을 그리지 않는다. 대신 **어떤 포맷으로, 어떤 표준으로, 어떤 순서로** 뽑을지 지휘하고, 실제 파일 생성은 전용 매뉴얼(pptx·docx·xlsx·pdf)을 불러 맡긴다.

## 왜 지휘와 생성을 나누는가

파일 포맷마다 잘하는 도구가 따로 있다. 슬라이드는 pptx, 문서는 docx, 표·계산은 xlsx, 고정 배포본은 pdf가 전담한다. 이 매뉴얼이 매번 파일 만드는 법을 새로 짜면 품질이 들쭉날쭉해진다. 그래서 **판단(무엇을·어떤 순서로)은 여기서, 실행(파일 만들기)은 전용 매뉴얼에서** 하도록 나눈다. 넘길 때는 그냥 넘기지 말고, 담을 내용과 템플릿 규격을 함께 넘겨라.

## 포맷 선택: 언제 무엇을 쓰는가

| 포맷 | 전담 매뉴얼 | 언제 쓰는가 | 대표 산출물 |
|---|---|---|---|
| PPT (덱) | pptx | 사람 앞에서 설득·발표할 때, 시각적 스토리 | 제안서, 캠페인 기획 발표 |
| Word (문서) | docx | 글로 읽고 회람할 때, 계약·상세 설명 | 리서치 보고서, 실행 가이드 |
| Excel (시트) | xlsx | 숫자를 계산·편집·시뮬레이션할 때 | 미디어 예산 배분표, KPI 트래커 |
| PDF (고정) | pdf | 그대로 배포·인쇄·서명할 때, 수정 불가 배포본 | 최종 제안서 배포본, 인쇄용 |

판단 기준:
- **바뀔 숫자가 있으면 Excel**. 예산·KPI처럼 클라이언트가 값을 바꿔볼 여지가 있으면 시트로 준다.
- **발표하면 PPT, 읽히면 Word**. 말로 설명을 얹으면 덱, 혼자 읽어야 하면 문서.
- **더 못 바꾸게 잠그면 PDF**. 최종본을 확정 배포할 때만.

Why: 미디어 플랜을 PDF로만 주면 클라이언트가 예산을 못 바꿔 답답하고, 발표 자료를 Word로 주면 지루하다. 내용의 쓰임새가 포맷을 결정한다.

## 팀 표준 템플릿 고정 원칙

같은 대행사에서 나온 문서는 하나처럼 보여야 한다. 매번 다른 색·폰트·레이아웃이면 아마추어로 보인다.

- **디자인 시스템을 고정한다**: 색(Ink Navy `#1E293B`, Signal Coral `#FF6B4A`, Fresh Teal `#2DD4BF`, Slate `#64748B`, Paper `#F8FAFC`), 표지·간지·본문 레이아웃, 제목 위계를 문서 종류마다 동일하게 적용.
- **표지 구성 고정**: 프로젝트명 · 클라이언트 · 날짜 · 대행사 로고 자리를 항상 같은 위치에.
- **표 스타일 고정**: 머리행 색, 정렬(숫자는 우측), 천단위 콤마를 통일.
- 전용 매뉴얼에 넘길 때 이 규격을 **명시적으로 함께 전달**하라. "표준 템플릿대로"라고만 하면 도구는 기본값으로 만든다.

Why: 표준을 고정하면 매번 디자인을 새로 정할 필요가 없고, 클라이언트는 "이 대행사 문서"라고 바로 알아본다. 일관성이 곧 전문성이다.

## 보고서 패키지 파이프라인 (핵심)

"한 번 요청 → 보고서 패키지"는 하나의 캠페인 결과를 **여러 포맷으로 동시에** 뽑아 한 묶음으로 전달하는 흐름이다. 예: 미디어 플랜 하나를 발표용 덱 + 상세 문서 + 편집 가능 시트 + 배포용 PDF로.

### 파이프라인 순서

1. **내용 확정**: 어떤 결과물을 문서화할지, 무엇을 담을지 먼저 정리한다. 내용이 흔들리면 4개 포맷을 4번 다시 만들게 된다.
2. **포맷 세트 결정**: 위 선택 기준으로 필요한 포맷을 고른다. 모든 요청이 4종 다 필요하진 않다 — 필요한 것만.
3. **표준 규격 부착**: 각 포맷에 팀 표준 템플릿 규격을 붙인다.
4. **전용 매뉴얼 순차 실행**: pptx → docx → xlsx → pdf 순으로 각 매뉴얼을 트리거해 파일을 만든다. 같은 내용을 넘기되 포맷 특성에 맞게 재구성한다(덱은 요약·시각, 문서는 상세).
5. **PDF는 마지막**: 배포용 PDF는 보통 덱이나 문서를 확정한 뒤 그것을 고정본으로 내보낸다. 그래서 순서상 끝이다.
6. **패키지 점검**: 4개 문서의 **핵심 숫자·메시지가 서로 일치**하는지 대조한다(예: 덱의 총예산과 시트의 합계). 이 교차 확인은 quality-review와 함께 반드시 거친다.

Why: 포맷마다 따로 만들면 덱과 시트의 숫자가 어긋나기 쉽다. 하나의 확정 내용에서 파생시키고 마지막에 대조하면 패키지 전체가 정합적이다.

## 산출물 형식

최종 산출은 실제 파일들(.pptx / .docx / .xlsx / .pdf)이다. 이 매뉴얼 자체의 출력은 "어떤 포맷을 어떤 내용·규격으로 만들지 정리한 지시"이며, 그 지시로 전용 매뉴얼을 호출한다. 파일을 직접 손으로 만들려 하지 마라 — 반드시 전용 매뉴얼에 위임한다.

## 품질 기준 (자가 점검)

- [ ] 내용의 쓰임새에 맞는 포맷을 골랐다(바뀔 숫자=Excel, 발표=PPT 등)
- [ ] 팀 표준 템플릿 규격(색·표지·표 스타일)을 전용 매뉴얼에 명시해 넘겼다
- [ ] 패키지의 경우 포맷 간 핵심 숫자·메시지가 일치한다
- [ ] 실제 파일 생성을 pptx/docx/xlsx/pdf 매뉴얼에 위임했다
- [ ] 배포용 PDF는 확정본을 기반으로 마지막에 만들었다

## 자주 하는 실수

1. **포맷을 잘못 고름** → 편집해야 할 예산표를 PDF로 주면 클라이언트가 못 고친다. 쓰임새를 먼저 물어라.
2. **표준 없이 기본값으로 산출** → 도구 기본 디자인은 밋밋하고 매번 다르다. 표준 규격을 함께 넘겨라.
3. **내용이 안 굳었는데 4종부터 만듦** → 내용을 먼저 확정하라. 나중에 고치면 4배 일이다.
4. **패키지 안 숫자가 서로 다름** → 덱 총예산과 시트 합계가 어긋나면 신뢰를 잃는다. 마지막에 교차 대조하라.
5. **직접 파일을 만들려 함** → 이 매뉴얼은 지휘자다. 실제 파일은 전용 매뉴얼이 만들어야 품질이 균일하다.

> AI는 같은 내용도 매번 조금씩 다르게 배치한다. 자동 생성된 문서는 그대로 보내지 말고, 표지 정보와 핵심 숫자를 사람이 한 번 확인한 뒤 배포하라.
