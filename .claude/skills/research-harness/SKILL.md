# Research Harness — 리서치·인사이트 하네스 (팬아웃/팬인)

하나의 카테고리를 **네 관점(시장·소비자·경쟁·트렌드)으로 동시에 조사(팬아웃)한 뒤 하나의 인사이트 보고서로 통합(팬인)** 하는 오케스트레이터(팀장)입니다. 여러 리서처가 각자 다른 각도를 병렬로 파서 속도와 관점 다양성을 동시에 확보하고, 인사이트 에디터가 교차검증하며 종합합니다.

**실행 모드:** 에이전트 팀(기본). 4개 조사는 서로 독립이므로 병렬로 돌리고, 통합 단계는 한 명이 모아 판단합니다. 모든 에이전트는 `model: opus`로 호출합니다.

## 팀 구성 (에이전트 풀에서)

| 팀원(에이전트) | 배정 관점 | 사용 매뉴얼(스킬) | 출력 |
|---|---|---|---|
| researcher(시장) | 시장 규모·성장성 | market-research (+deep-research) | `_workspace/1_researcher_market.md` |
| researcher(소비자) | 소비자 인식·페인포인트 | market-research | `_workspace/1_researcher_consumer.md` |
| researcher(경쟁) | 경쟁사 포지셔닝 | market-research | `_workspace/1_researcher_competitor.md` |
| researcher(트렌드) | 최신 트렌드 | market-research | `_workspace/1_researcher_trend.md` |
| insight-synthesizer | 교차검증·종합(팬인) | insight-synthesis | `_workspace/2_insight_report.md` |

## Phase 0: 컨텍스트 확인 (초기/후속/부분 재실행 판별)

1. `_workspace/` 존재 여부를 확인한다.
2. 실행 모드 결정:
   - **없음** → 초기 실행. Phase 1로.
   - **있음 + 특정 관점 수정 요청**(예: "경쟁 조사만 다시", "트렌드 업데이트") → 부분 재실행. 해당 관점 리서처 1명만 재호출해 그 파일만 덮어쓰고, 이어서 insight-synthesizer만 재실행해 보고서를 갱신한다.
   - **있음 + 새 카테고리** → 새 실행. 기존 `_workspace/`를 타임스탬프 폴더로 이동 후 새로 시작.
3. 부분 재실행 시 나머지 3개 조사 파일은 보존해 통합에 재사용한다.

## Phase 흐름 (팬아웃 → 팬인)

### Phase 1: 준비
1. 브리프에서 조사 대상 **브랜드/카테고리, 기간, 지역** 스코프를 고정한다(범위가 넓으면 조사가 얕아짐).
2. `_workspace/` 생성, 입력을 `_workspace/00_input/brief.md`로 저장.

### Phase 2: 팀 구성
1. `research-team` 을 만들고 리서처 4명 + insight-synthesizer 1명을 `model: opus`로 등록한다.
2. 작업 등록: 4개 조사 작업(관점별)은 서로 의존이 없다. 통합 작업 1개는 4개 조사 작업 모두에 `depends_on`으로 연결한다.

### Phase 3: 팬아웃 — 4관점 병렬 조사
**실행 방식:** 팀원 자체 조율. 4명이 공유 작업 목록에서 각자의 관점 작업을 요청(claim)해 **동시에** 진행한다.
- 각 리서처는 배정된 한 관점만 판다(관점을 섞으면 출처가 엉킨다).
- 모든 주장에 출처를 병기하고, 확인 불가 항목은 솔직히 남긴다.
- 관점별 결과를 `_workspace/1_researcher_{관점}.md`로 저장한다.

### Phase 4: 팬인 — 교차검증·종합
1. insight-synthesizer가 4개 조사 파일을 모두 Read한다.
2. 교차검증: 여러 관점이 같은 결론이면 신뢰도↑로 표기, **상충하면 출처를 병기**(한쪽을 지우지 않는다).
3. "주장 → 근거 → 시사점" 구조로 **캠페인 방향 인사이트 3개**를 도출한다.
4. 결과를 `_workspace/2_insight_report.md`로 저장한다.

### Phase 5: 마무리
1. 최종 보고서를 지정 경로 `course2-research/사례/C2_제로톡_카테고리진단.md` 로 내보낸다.
2. 필요 시 doc-producer로 Word/PDF 변환을 이어붙일 수 있음을 안내한다(doc-automation).
3. 팀 정리, `_workspace/` 보존.

## 데이터 전달

- **팀 모드:** 태스크 + 파일 + 메시지. 통합 담당(insight-synthesizer)이 4개 조사 파일 경로를 받아 Read로 수집한다.
- **파일 규약:** `_workspace/{phase}_{agent}_{artifact}.{ext}`. 관점 구분을 위해 artifact 자리에 관점명을 쓴다(market/consumer/competitor/trend).
- **최종 산출물만** `course2-research/사례/…`로 내보낸다.

## 데이터 흐름(그림)

```
                    ┌─▶ [researcher 시장]  ──▶ 1_researcher_market.md ─┐
[팀장] → 팀 구성 ──┼─▶ [researcher 소비자]──▶ 1_researcher_consumer.md┤
   (팬아웃, 동시)   ├─▶ [researcher 경쟁] ──▶ 1_researcher_competitor.md│
                    └─▶ [researcher 트렌드]──▶ 1_researcher_trend.md ───┤
                                                          (팬인) Read ◀─┘
                                                              ▼
                              [insight-synthesizer] 교차검증·종합
                                                              ▼
                              course2-research/사례/C2_제로톡_카테고리진단.md
```

> **비결정성 안내:** 병렬 조사는 매 실행마다 찾는 근거가 조금씩 달라질 수 있습니다. 재현 편차는 정상이며, 핵심은 "근거와 출처가 붙어 있는가"입니다.
> **검증 습관:** 여러 관점이 같은 결론이면 신뢰하고, 상충하면 출처를 병기해 두 값을 모두 남깁니다. 한쪽을 임의로 지우지 마세요.

## 에러 핸들링

| 상황 | 대응 |
|---|---|
| 리서처 1명 조사 실패 | 방식 바꿔 1회 재시도 → 실패 시 해당 관점 "확인 불가" 명시, 나머지 3관점으로 통합 진행 |
| 관점 간 상충 데이터 | 삭제 금지, 출처를 나란히 병기하고 통합 단계에서 신뢰도 표기 |
| 조사 범위 과다로 얕음 | 스코프(기간·지역·카테고리)를 좁혀 재실행 |
| 근거 없는 '느낌' 인사이트 | 반드시 조사 파일의 근거로 연결, 근거 없으면 인사이트에서 제외 |

## 팀 크기

**중규모(5명).** 관점 4개가 병렬성의 핵심이므로 리서처 4명을 유지한다. 관점을 늘리면(예: 유통·가격) 리서처를 추가하되 통합 담당은 1명으로 유지한다.

## 테스트 시나리오

### 정상 흐름
1. 사용자가 "제로톡" 이 속한 제로 슈거 음료 카테고리 진단을 요청.
2. Phase 1에서 스코프(국내, 최근 2년) 고정.
3. Phase 3에서 리서처 4명이 시장/소비자/경쟁/트렌드를 병렬 조사 → 4개 파일 저장.
4. Phase 4에서 insight-synthesizer가 교차검증 후 인사이트 3개 도출.
5. 최종 보고서 `course2-research/사례/C2_제로톡_카테고리진단.md` 생성.

### 에러 흐름
1. Phase 3에서 경쟁 리서처가 특정 경쟁사 점유율을 못 찾음.
2. 1회 재시도 후에도 실패 → "경쟁사 점유율 확인 불가"로 명시.
3. insight-synthesizer는 나머지 근거로 통합하되, 해당 인사이트에 "근거 일부 미확보" 라벨.
4. 최종 보고에 미확보 영역 명시.

## 팀장 완주 체크

- [ ] 조사 스코프(브랜드·기간·지역)를 고정했는가
- [ ] 4개 관점이 **병렬로** 조사되고 각각 파일로 저장됐는가
- [ ] 모든 주장에 출처가 붙었고, 상충 데이터는 병기됐는가
- [ ] 인사이트 3개가 "주장→근거→시사점" 구조인가
- [ ] 최종 보고서만 `course2-research/사례/`로 내보냈는가

## 실습 트리거 대표 프롬프트

```text
리서치 하네스를 구성해줘. 제로 슈거 음료 카테고리(우리 브랜드 "제로톡" 포함)를 여러 각도에서 동시에 조사하는 팀이 필요해 — 시장 규모와 성장성, 소비자 인식과 페인포인트, 경쟁사 포지셔닝, 최신 트렌드. 각자 조사한 뒤 교차검증하고, 상충하는 정보는 출처를 함께 남겨서 '캠페인 방향 인사이트 3개 + 근거'가 담긴 종합 보고서를 만들어줘.
```

후속 예: "경쟁 파트만 다시 조사해줘" · "트렌드를 최신으로 업데이트" · "이전 보고서 기반으로 인사이트 개선".
